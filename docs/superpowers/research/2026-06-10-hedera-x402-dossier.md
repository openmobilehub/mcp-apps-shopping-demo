## Bottom line

Path 1 (native Ed25519 Hedera account, StrongBox key IS the account key) is the correct architecture and is recommended, gated on one on-device check that StrongBox actually holds Ed25519 on your Pixel. The chain, the merged native x402 exact scheme, and the live blocky402 testnet facilitator all support it; Path 2 (secp256k1 EVM facade plus EIP-3009) is not viable because canonical Circle USDC on Hedera is a native HTS token with no EIP-3009 on its EVM facade.

## Synthesis

### Path 1 (native Ed25519) viability

VIABLE and recommended, conditional on one on-device check. Hedera natively accepts Ed25519 as a first class account key (original native curve; ECDSA added via HIP-222); a single Ed25519 key signs a native TransferTransaction for HBAR or HTS with no contract or facade. The exact scheme is merged into coinbase x402 (PR 792, 2026-02-06). The blocky402 signer hands a caller supplied key to the SDK sign call with zero curve checks; all nine unit tests use the Ed25519 generator including the fee payer. Live supported confirms scheme exact, hedera testnet, feePayer 0.0.7162784, HTTP 200. The Hedera Java SDK exposes signWith and addSignature so the key stays out of process; the Ed25519 signature pair wants a raw 64 byte R then S value, which the Multipaz cose encoded signature produces. Not confirmed (see blockers): StrongBox Ed25519 on your device; live verify acceptance of an Ed25519 signed tx (every published example is ECDSA); a Multipaz keystore Ed25519 return path bug. Path 2 is not viable for canonical Circle USDC on Hedera, a native HTS token (testnet 0.0.429274) whose EVM facade exposes only standard ERC-20 with no transferWithAuthorization. Net: Path 1 is structurally clean (no smart wallet, bundler, paymaster, or secp256k1 seed in RAM window) and is the path Hedera paves.

### Recipient-binding (Narb's anti-dishonest-facilitator mechanism)

The primitive is the ordinary native Hedera CryptoTransfer (TransferTransaction) used as a partially signed transaction, not Scheduled Transactions, not allowances, not EIP-3009. Verified by decompiling the package and reading the merged spec. The client builds a TransferTransaction crediting payTo for exactly the required amount and sets the transaction id account to the facilitator fee payer, then freezes and signs. The Hedera signature covers the body bytes (the serialized transaction body, which the protobuf says is what the signature map signs), and recipient plus amount live inside that body. The facilitator can only append its fee payer signature and broadcast, or refuse, because any edit to payTo or amount changes the body bytes and invalidates the client signature, so nodes reject with an invalid signature error; the signature map is append only. This is equivalent to EIP-3009 binding destination and value, but cleaner since the signed artifact is itself a ledger transaction. Defense in depth, the spec must rules in verify: reject net credit to payTo not equal to amount, reject any positive receiver other than payTo, reject transaction id account not equal to fee payer, reject non transfer ops, reject fee payer as net sender. Caveats: one verdict downgraded to partly the exact phrasing that mutating payTo or amount breaks the signature (true from the protobuf, but the spec stated defense is the must rules); and the spec stated motivation is protecting the fee payer while Narb frames it as protecting the recipient. Both protections are real.

### Recovery story

Definitively yes: Hedera AccountUpdate key rotation and threshold or key list keys together resolve the single point of failure of a non extractable StrongBox key, natively, with no smart contract. AccountUpdate rotates the account key but must be signed by both the old key (authorization) and the new key (safeguard). The account key can itself be an M of N threshold key or key list used directly, since the protobuf key is a recursive choice of ed25519, secp256k1, key list, or threshold key, and leaves can mix curves, so the key can be a two of three over the phone StrongBox Ed25519 plus two recovery keys. Honest limit: a pure single StrongBox key with no backup share is still a hard single point of failure, because after device loss the old key is gone and AccountUpdate requires it, so you cannot self rotate. The fix is to provision a threshold key from day one (routine payments still require the phone, recovery uses the backup quorum if the phone dies) or rotate into a threshold while you still hold the original key. Caveats: mixing Ed25519 and ECDSA leaves is permitted at the spec level (verdict likely) but lacks a worked node acceptance example, so confirm on testnet; where the non phone shares live is a trust decision; and whether x402 tooling accepts a threshold keyed payer is unconfirmed. Net: Path 1 native recovery is at least as strong as the multi owner Coinbase Smart Wallet model your PATH_A_NEXT.md designed for EVM, with no contract, so do not ship a single key account.

### Signature model

Two signatures, not one, by construction, but both drivable from one biometric or one Multipaz getCredential unlock. The OID4VP or caBLE artifact and the Hedera settlement artifact sign different payloads. The OID4VP holder proof (key binding JWT or mdoc device signature) signs a JOSE or COSE payload with nonce, audience, issued at, disclosure hash, and the transaction data hashes; the amount and payee binding rides in the transaction data hashes, a hash of the intent, an off chain AP2 style mandate, not a submittable transaction. Hedera settlement signs the body bytes, the protobuf transaction body. Different encodings, different bytes, different signature objects, so OID4VP transaction data is the wrong vehicle to carry the chain transaction. caBLE: the W3C Digital Credentials API cross device flow carries the OID4VP presentation over the CTAP or Bluetooth channel, the mandate, not the transaction. The Hedera transaction is built by the agent, its body bytes go to the phone per HIP-179 external transaction signing, where the signer returns only a signature for the signature map and the transaction must equal the request; the phone StrongBox key signs the body bytes under the same biometric. Curve complication: a HAIP compliant OID4VP leg mandates ES256 P-256 as the floor (EdDSA permitted but not required), so the key binding JWT key is naturally StrongBox P-256, while the Hedera key must be Ed25519 or secp256k1, not P-256. So the two keys are different curves: one biometric, two key handles. Caveat: for ISO mdoc the transaction data binding placement (18013-7) is less settled than the SD-JWT VC path (OID4VP issue 519), so prefer SD-JWT VC for the authorization leg.

### Blockers / unconfirmed

- StrongBox Ed25519 on your device is the dossier one real contradiction: Thread 6 says P-256 only while your SecpStrongboxProbe and Thread 7 say Ed25519 works at KeyMint 2.0 plus. Re-confirm on-device that the security level reads STRONGBOX not TEE.
- Multipaz keystore Ed25519 return path bug: the keystore sign routes Ed25519 through a DER decode but the platform returns raw R then S, so it may throw on-device; mitigate by using the cose encoded signature and self verifying before submit.
- Facilitator Ed25519 acceptance is not proven end to end: source is curve agnostic and unit tested on Ed25519, but every published example is ECDSA and live verify was not exercised with Ed25519.
- Frozen transaction expiry is tight for async approval at roughly 120 to 180 seconds; Scheduled Transactions HIP-423 fix it but the shipping scheme uses frozen bytes. The window figure was not re-verified from primary docs.
- HTS association: a fresh Ed25519 account must associate with USDC 0.0.429274 before holding it; native HBAR asset 0.0.0 avoids it.
- aliasPolicy and hollow account: a payTo given as an alias can trigger facilitator funded auto account creation; the facilitator defaults to reject, so keep a provisioned 0.0.x id.
- Low severity nits: confirm the x402 version is 2; pin a known good package artifact; mainnet is coming soon and testnet only, which fits your target.
- One dossier claim was refuted: that Hedera x402 is just an Agent Kit plugin and not a wire scheme is false; it is a formal native wire scheme merged into x402, which strengthens Path 1.

### Thread: Can the OID4VP/caBLE amount-binding signature double as the Hedera settlement authorization, and should the wallet use a native Ed25519 Hedera account (Path 1) or a secp256k1 EVM facade (Path 2)?

**Takeaway:** No — a single OID4VP presentation cannot double as the Hedera settlement signature, because the two artifacts sign structurally different payloads: the OID4VP Key Binding JWT signs a JOSE object {nonce, aud, sd_hash, transaction_data_hashes}, while Hedera settlement signs the protobuf-serialized TransactionBody bodyBytes. They are two signatures over two payloads by construction. They CAN, however, both be produced from one Multipaz getCredential / one biometric unlock if (and only if) the same key/curve is usable for both — and that is where the design breaks. The most important finding, which cuts across both Path 1 and Path 2: Hedera accounts accept ONLY Ed25519 and ECDSA(secp256k1) keys, while Android StrongBox (Titan M2 on Pixel) supports ONLY NIST P-256 (secp256r1) for EC and does NOT support Ed25519. Those curve sets are disjoint, so a raw StrongBox-resident EC key cannot itself be a Hedera account key in EITHER path on a Pixel. Path 1's premise ("the StrongBox Ed25519 key IS the account key") is therefore not achievable on a Titan-M2 Pixel today: StrongBox won't hold an Ed25519 key. The realistic design is two signatures from the same biometric: (a) an OID4VP presentation as the AP2-style authorization mandate (signed by a StrongBox P-256 KB-JWT key — HAIP-compliant, ES256), and (b) a separate Hedera bodyBytes signature, whose key must live in a software/TEE SecureArea (Ed25519 or secp256k1) rather than StrongBox, unless you accept the P-256 KB-JWT key as authorization-only and a different key as the on-chain settlement key. Narb's recipient-binding claim is real and standard: the Hedera x402 'exact' scheme has the client sign a partially-signed transfer whose recipient+amount live inside the signed bodyBytes, so the facilitator can only broadcast, not redirect — exactly analogous to EIP-3009's signed {to, value} in the EVM scheme.

### Thread: Can a Multipaz SecureArea Ed25519 key sign a Hedera transaction in practice? Path 1 (native Ed25519 Hedera account = StrongBox key) vs Path 2 (secp256k1 EVM facade + EIP-3009).

**Takeaway:** Yes — the mechanism is sound and the integration seam exists on both sides, with one concrete code-level caveat to validate first. The Hedera Java/Kotlin SDK exposes exactly the external-signer hooks needed: Transaction.signWith(PublicKey, UnaryOperator<byte[]>) hands your callback the transaction BODY bytes and lets it return a precomputed signature, and Transaction.addSignature(PublicKey, byte[]) attaches an already-computed signature — so the private key never leaves StrongBox. The bytes to sign are the TransactionBody protobuf bytes (after freezeWith), and Hedera's SignaturePair.ed25519 field expects a RAW 64-byte (R||S) Ed25519 signature, which is exactly what Multipaz's EcSignature.toCoseEncoded() (= r + s) produces. THE CAVEAT: Multipaz's AndroidKeystoreSecureArea.sign() runs every algorithm — including Ed25519 — through signatureFromDer(), which ASN.1-decodes the bytes, but a JCA Ed25519 signer returns raw R||S, not DER (Multipaz's own software path and its Crypto.sign KDoc confirm Ed25519 is raw). So the in-enclave Ed25519 return path looks buggy/untested in the AndroidKeystore implementation and must be validated on-device (the software SecureArea path is correct). Separately, for the architecture decision: blocky402's published Hedera example uses ECDSA (secp256k1) keys and an ExactHederaScheme, and Hedera's own x402 scheme is a partially-signed NATIVE Hedera transaction (client signs the transfer, facilitator only pays gas and submits) — that native partial-signing is the actual mechanism behind Narb's dishonest-facilitator protection. That property holds for BOTH an Ed25519 and an ECDSA Hedera account; it is NOT a reason to prefer secp256k1, and it is unrelated to EIP-3009.
