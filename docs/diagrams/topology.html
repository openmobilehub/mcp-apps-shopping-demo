Your server is both a server (to the agent) and a client (to the merchant). The gate cleanly intercepts complete_checkout, so the merchant is a pluggable adapter behind one interface, the same pattern as the settlement rails. Best fit for the clone-and-run demo.
MCP server topology · option B

Agent orchestrates both as peers (siblings)

No wrapping. The agent connects to the merchant’s Checkout MCP and your MCP server directly. Yours is the authorization / credential capability the host invokes at the pay step.

The two servers are peers, not nested. Purer MCP and likely how the ecosystem composes long-term, but it depends on the host willing to orchestrate two servers and invoke your gate at exactly the right moment, less control and harder to guarantee on stage.
